# PRD: Stable Database IDs

## Overview

This feature refactors database relationships so application behavior no longer depends on database-generated serial row IDs as foreign keys. Stable IDs should be generated or derived outside of database insertion order, remain indexable, and support safer migration, import/export, and environment synchronization.

The work covers the full current data model. It includes schema updates, DAO/service/query updates, tests, and a production data migration run from the local/admin environment. Migration execution code does not need to be deployed with the hosted application.

## Goals

1. Replace generated row-id foreign key relationships with stable IDs across the database.
2. Use stable identifiers that are compact, indexable, and independent of row creation order.
3. Preserve or derive stable IDs from existing natural/domain identifiers where available.
4. Avoid inventing row IDs for join/state tables whose rows are already uniquely defined by their related stable keys.
5. Migrate existing local and production data safely using local/admin tooling.

## Scope Boundaries

## In Scope

- Stable ID design for all existing relational entities.
- Schema changes for stable primary keys, unique keys, and foreign key relationships.
- DAO, reader, service, route, importer, and test updates needed to use stable IDs.
- Videos to source channels.
- Virtual channel assignments to source and virtual channels.
- Assignment video selections to assignments/videos or equivalent stable composite keys.
- Flags and watch history to videos/profiles.
- Production migration run from this environment using the admin database user.
- Removal of obsolete generated-row-ID assumptions from code and tests.

## Out of Scope

- Deploying migration execution code to the hosted app.
- Keeping generated database row IDs as the relationship contract.
- Requiring UUID-sized identifiers unless a specific implementation constraint forces it.
- Adding standalone IDs to tables whose rows are already well-defined by their foreign key tuple.

## Stable ID Requirements

1. Stable IDs must not be generated by auto-increment columns or otherwise depend on insert order.
2. Stable IDs must be short enough and typed appropriately for efficient indexing in MySQL/MariaDB.
3. YouTube-backed entities should derive stable IDs from YouTube IDs where available:
   - Source channels from `source_channels.youtube_id`.
   - Videos from `videos.youtube_id`.
4. Local entities with natural keys should derive stable IDs from those natural keys where practical:
   - Profiles from `profiles.key`.
   - Virtual channels from `virtual_channels.name`, unless implementation discovers a safer existing key.
5. Tables representing state for a natural relationship should use the related stable keys as their identity instead of adding a new synthetic stable ID when the tuple already defines uniqueness.
6. Existing auto-increment IDs may remain temporarily or privately if useful during migration, but app relationships must not rely on them after the feature is complete.

## Data Model Requirements

1. `source_channels` must expose a stable channel identifier suitable for references from videos and assignments.
2. `videos` must reference source channels by stable channel identifier rather than generated row ID.
3. `virtual_channels` must expose a stable identifier suitable for assignment references.
4. `virtual_channel_assignments` must reference source and virtual channels by stable identifiers.
5. `virtual_channel_assignment_video_selections` must be identifiable by its related assignment/video stable relationship rather than a generated row ID where practical.
6. `profiles` must expose a stable profile identifier suitable for flags and watch history.
7. `video_flags` must be keyed by stable video/profile identifiers.
8. `watch_history` must reference stable video/profile identifiers. Individual history sessions may keep an implementation row ID if needed, because a session is not uniquely defined only by video/profile.
9. Foreign keys and indexes must be updated to match the new relationship model.

## Migration Requirements

1. Provide a local/admin migration path for the existing production database.
2. The migration must preserve all current data and relationships.
3. The migration must be runnable from this workspace using admin credentials, not by deployed app startup code.
4. The migration should be inspectable before execution, preferably as SQL or a small script that emits/logs the SQL steps.
5. The migration must handle existing records deterministically:
   - YouTube-backed stable IDs derive from existing YouTube IDs.
   - Natural-key stable IDs derive from existing profile keys and virtual channel names where applicable.
   - Relationship tables are rewritten using joins from existing generated IDs to the new stable identifiers.
6. After migration, production app credentials should only need normal runtime permissions, not schema migration permissions.

## Application Requirements

1. DAOs and readers must accept and return stable identifiers where they currently expose relationship row IDs.
2. Services and route handlers must stop depending on generated row IDs for cross-table relationships.
3. YouTube import/update paths must create or use stable IDs consistently.
4. Viewer/admin actions must continue to behave the same from the user perspective.
5. Tests must cover the updated ID behavior at DAO/query/service boundaries.
6. Existing behavior for flags, history, virtual channel assignments, and video selection must remain intact.

## Validation Requirements

1. Run the standard project validation:
   - `npm run check`
   - `npm run typecheck`
   - `npm run test`
2. Add or update focused tests proving:
   - SQL uses stable identifiers for key relationships.
   - Join/state tables use composite stable keys where appropriate.
   - Import paths preserve stable IDs for YouTube-backed entities.
   - Migration logic preserves relationships.
3. Before production migration, verify the migration against local/dev data or a representative export.
4. After production migration, verify core app workflows against production:
   - viewer loads videos
   - history records
   - flags update
   - virtual channel assignments work
   - selected video review state persists

## Constraints and Considerations

- This feature should prefer the simplest stable ID scheme that is indexable and deterministic enough for the data model.
- Opaque strings are acceptable; human readability is not required.
- Deterministic IDs based on existing natural keys are preferred where they reduce migration complexity.
- The app should not require deployed migration code for this feature.
- The production migration should be treated as an operational step with admin credentials.
- If an existing auto-increment column is left in place, it should be treated as private/internal and not used as a relationship contract.

## Success Criteria

1. No application relationship depends on database-generated serial row IDs.
2. Current user-facing behavior is unchanged.
3. Existing data is migrated without relationship loss.
4. Runtime production credentials do not need schema migration privileges.
5. Project checks and tests pass.
