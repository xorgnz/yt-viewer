# Database Change Plan: Stable Database IDs

## Status

Draft for task 2.1 through task 2.3. This file proposes the exact table, column, key, index, foreign key, stable ID source, and production migration changes. It is not approved for schema edits, migration scripts, or production database changes until task 2.4 receives explicit user approval.

## Design Direction

- Keep YouTube-backed stable IDs as the existing YouTube IDs.
- Keep profile stable IDs as the existing profile keys.
- Add an immutable stable key for virtual channels so renaming a virtual channel does not change relationship identifiers, URLs, or saved filters.
- Remove generated row IDs from application relationship contracts.
- Keep private generated row IDs only where they identify implementation events rather than relationships.

## Stable ID Sources

| Entity/table | Stable ID column or identity | Source rule | Existing data backfill rule |
| --- | --- | --- | --- |
| `source_channels` | `youtube_id` | Use the YouTube channel ID returned by YouTube APIs or entered by the admin. | Reuse existing `source_channels.youtube_id`. Rows with missing or duplicate values should block migration. |
| `videos` | `youtube_id` | Use the YouTube video ID returned by playlist/video metadata. | Reuse existing `videos.youtube_id`. Rows with missing or duplicate values should block migration. |
| `profiles` | `key` | Use the app profile key managed by `ProfileCatalog`, such as `adult` or `child`. | Reuse existing `profiles.key`. Rows with missing or duplicate values should block migration. |
| `virtual_channels` | `key` | Generate once from `name` using a deterministic slug rule at creation time; store it as immutable app-managed identity. | Backfill from existing `virtual_channels.name` using the same slug rule, with collision suffixes if needed. |
| `virtual_channel_assignments` | `(source_channel_youtube_id, virtual_channel_key)` | Use the stable source-channel and virtual-channel identifiers. | Join current `source_channel_id` to `source_channels.id` and `virtual_channel_id` to `virtual_channels.id`. |
| `virtual_channel_assignment_video_selections` | `(source_channel_youtube_id, virtual_channel_key, video_youtube_id)` | Use the stable assignment tuple plus stable video ID. | Join current `assignment_id` to `virtual_channel_assignments.id`, then join source/virtual/video generated IDs to stable IDs. |
| `video_flags` | `(video_youtube_id, profile_key)` | Use stable video ID plus stable profile key. | Join current `video_id` to `videos.id` and `profile_id` to `profiles.id`. |
| `watch_history` relationships | `video_youtube_id`, `profile_key` | Use stable video ID and stable profile key; keep `id` for private session row identity. | Join current `video_id` to `videos.id` and `profile_id` to `profiles.id`. |
| `migration_history` | `id` | Keep generated ID as internal migration bookkeeping. | No change. |
| `_meta` | `key` | Existing stable metadata key. | No change. |

## Virtual Channel Key Rule

`virtual_channels.key` should be immutable after creation. It should be generated by application code, not by the database.

Proposed generation rule:

1. Trim the display name.
2. Convert to lowercase.
3. Replace each run of non-alphanumeric characters with `-`.
4. Trim leading and trailing `-`.
5. Use `virtual-channel` if the slug would otherwise be empty.
6. If the slug already exists, append `-2`, `-3`, and so on until unique.

Existing rows should be backfilled in deterministic order by current generated `id` so collision suffixes are reproducible. After migration, renaming `virtual_channels.name` must not change `virtual_channels.key`.

## Composite Key Rules

| Table | Composite identity | Rationale |
| --- | --- | --- |
| `virtual_channel_assignments` | `(source_channel_youtube_id, virtual_channel_key)` | The assignment is naturally one source channel associated to one virtual channel. |
| `virtual_channel_assignment_video_selections` | `(source_channel_youtube_id, virtual_channel_key, video_youtube_id)` | Review state is naturally one video under one assignment. |
| `video_flags` | `(video_youtube_id, profile_key)` | Flag state is naturally one video per profile. |

No standalone stable ID should be added for these rows unless implementation later finds a MySQL/MariaDB constraint that makes the composite key unsafe or impractical.

## Production Migration Sequence

The migration should be created as an inspectable local/admin artifact in task 6, not deployed as app startup code.

Planned sequence:

1. Take a production database backup using the hosting control panel or `mysqldump` before any schema changes.
2. Record current row counts for all affected tables: `source_channels`, `videos`, `profiles`, `virtual_channels`, `virtual_channel_assignments`, `virtual_channel_assignment_video_selections`, `video_flags`, and `watch_history`.
3. Run preflight checks for null or duplicate stable source values:
   - `source_channels.youtube_id`
   - `videos.youtube_id`
   - `profiles.key`
   - generated virtual-channel keys after slugging `virtual_channels.name`
4. Add new stable relationship columns as nullable side-by-side columns.
5. Backfill direct stable IDs:
   - `virtual_channels.key` from `virtual_channels.name`
   - `videos.channel_youtube_id` from `source_channels.youtube_id`
   - `virtual_channel_assignments.source_channel_youtube_id` from `source_channels.youtube_id`
   - `virtual_channel_assignments.virtual_channel_key` from `virtual_channels.key`
   - `video_flags.video_youtube_id` from `videos.youtube_id`
   - `video_flags.profile_key` from `profiles.key`
   - `watch_history.video_youtube_id` from `videos.youtube_id`
   - `watch_history.profile_key` from `profiles.key`
6. Backfill `virtual_channel_assignment_video_selections` stable tuple columns by joining through the current generated assignment and video IDs.
7. Validate every new stable column is non-null and row counts still match the pre-migration counts.
8. Add stable unique constraints, indexes, and foreign keys.
9. Update application code and tests to use the stable columns.
10. After stable app validation passes against migrated data, remove old generated relationship columns and obsolete indexes/foreign keys.
11. Keep only private generated `watch_history.id` and `migration_history.id`.
12. Run post-migration smoke checks against production.

## Backup and Rollback Expectations

Backup expectation:

- A complete production database backup must exist before the migration runs.
- The backup location and timestamp should be recorded in the migration execution notes.
- The migration script should print or log the database name, affected table list, and preflight counts before applying changes.

Rollback expectation:

- Before old generated columns are removed, rollback can use the same database state because both generated and stable columns exist side by side.
- After old generated columns are removed, rollback means restoring the pre-migration database backup.
- Do not run the destructive cleanup phase until the migrated schema has passed local/dev verification and the app has been updated to use stable IDs.

## Verification Expectations

Pre-cleanup verification:

- Confirm affected table row counts match pre-migration counts.
- Confirm every old generated relationship has a matching stable relationship.
- Confirm no stable relationship column is null.
- Confirm foreign key checks pass.

Representative workflow checks:

- Viewer loads videos.
- Viewer channel and virtual-channel filters work.
- Watch history records and updates a session.
- Watched/favorite/ignored flags update.
- Admin source-channel refresh still links videos to source channels.
- Admin virtual-channel assignments work.
- Selected-only review state persists.

## Approval Gate

No schema edits, migration scripts, or production database changes should begin until the user explicitly approves this plan.
