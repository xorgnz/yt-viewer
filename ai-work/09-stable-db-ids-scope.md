# Project Scope: Stable Database IDs

## Overview

Refactor the database model so relationships no longer depend on database-generated serial row IDs as foreign keys. Stable IDs should be generated by application/domain logic, remain independent of insertion order, and be safe to preserve across migrations, imports, exports, and production setup.

This feature should cover the full current data model and include a production database migration run from the local/admin environment. Migration logic does not need to be shipped as deployed application code.

## Problem Statement

Current table relationships use generated numeric row IDs. Those IDs are brittle because they depend on creation order and database-local state, making data migration, reseeding, import/export, and environment synchronization more fragile than necessary.

## Target Users

The primary users are the application maintainer and deployment operator. End users benefit indirectly through safer database evolution and reduced risk of broken relationships during future changes.

## Core Objectives

1. Replace generated-row-ID foreign key relationships with stable application/domain IDs across the database.
2. Update DAOs, readers, services, tests, and schema definitions to use stable IDs consistently.
3. Provide and run a production-safe migration from the local/admin environment using the admin database user.

## In Scope

- Stable ID design for all existing relational entities.
- Videos to source channels.
- Virtual channel assignments to source and virtual channels.
- Assignment video selections to assignments and videos.
- Flags and watch history to videos and profiles.
- Schema updates for stable primary or foreign key usage.
- Data migration SQL/scripts run locally against production with admin credentials.
- Cleanup of obsolete generated-ID assumptions in code and tests.

## Explicitly Out of Scope

- Deploying migration execution code to the hosted application.
- Preserving generated database row IDs as the relationship contract.
- Over-engineering IDs as UUIDs unless a specific table genuinely needs that shape.

## Assumptions and Constraints

- Stable IDs do not need to be globally huge or UUID-sized; they only need to be generated outside the database and independent of creation order.
- Existing auto-increment IDs may remain as private implementation details if that is simpler, but application relationships should not depend on them.
- Production migration will be run manually/from here using the admin user, not by the deployed app.
- The feature applies to the full current schema unless a later implementation detail proves a narrower exception is safer.
